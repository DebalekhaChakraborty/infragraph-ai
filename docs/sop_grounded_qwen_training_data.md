# SOP-Grounded Qwen Training Data

Supervised fine-tuning dataset for InfraGraph AI remediation behavior alignment.

---

## Purpose

This dataset trains Qwen to generate remediation plans that:
- Follow the InfraGraph AI remediation output schema exactly
- Cite KB-* and CE-* evidence IDs from the supplied context
- Respect domain-specific SOP constraints (load balancer, database, WAN, firewall)
- Never invent commands, IP addresses, owners, or device facts

The training target is the **deterministic template remediation output** for each scenario — a structured, SOP-grounded plan generated by `template_mode.py` from the same context supplied to the model.

This is **behavior alignment**, not knowledge injection. The SOP knowledge lives in the KB vector index; the model learns the correct format, citation discipline, and safety posture.

---

## What this dataset is NOT

- It is not a knowledge base. Facts about devices, topologies, and SOPs come from retrieved KB evidence in the user message, not from model weights.
- It is not a replacement for the KB index. Adding new SOPs requires KB re-indexing, not model retraining.
- It is not an evaluation dataset. No labels, ground truth, or evaluation fields appear anywhere.

---

## When to re-index vs when to retrain

| Change | Action |
|--------|--------|
| New SOP or runbook added to `assets/kb/` | Rebuild KB index only: `python scripts/build_kb_index.py --reset` |
| Existing SOP content revised | Rebuild KB index only |
| New alert type or node type added | Rebuild KB index if SOPs cover it; retrain only if model ignores new structure |
| Model outputs wrong schema fields | Retrain with updated training data |
| Model ignores KB-* citation discipline | Retrain |
| Model invents device names or commands | Retrain |
| More scenario coverage needed | Generate more scenarios, rebuild dataset, retrain |

## Record Format

Each JSONL record has three messages:

```json
{
  "id": "sop_grounded_enterprise_v3_0000",
  "messages": [
    {
      "role": "system",
      "content": "You are InfraGraph AI Remediation Agent. ..."
    },
    {
      "role": "user",
      "content": "<compact JSON incident context>"
    },
    {
      "role": "assistant",
      "content": "<strict JSON remediation output>"
    }
  ],
  "metadata": {
    "scenario_id":           "enterprise_v3_0000",
    "root_cause":            "DC-FW-01",
    "root_cause_diagram":    "datacenter_topology",
    "rca_source":            "Enterprise GNN RCA",
    "cluster_id":            "CLU-ent_enterprise_v3_0000-001",
    "cluster_score":         0.8273,
    "kb_evidence_count":     5,
    "causal_evidence_count": 5,
    "target_source":         "template_sop_grounded"
  }
}
```

### User content fields

| Field | Description |
|-------|-------------|
| `incident_id` | `INC-<scenario_id>` |
| `scenario_id` | e.g. `enterprise_v3_0000` |
| `scope` | Always `enterprise` for current dataset |
| `root_cause` | Predicted root-cause node ID |
| `root_cause_diagram` | Topology diagram containing root cause |
| `impacted_nodes` | Node IDs in the impact path |
| `impacted_diagrams` | All affected topology diagrams |
| `alert_timeline` | Observed alert events (no labels) |
| `candidate_ranking` | Top GNN RCA candidates with scores |
| `cluster_id` | Event correlation cluster ID |
| `cluster_score` | Cluster confidence score |
| `correlation_reasons` | Why events were correlated |
| `causal_evidence` | CE-* causal evidence items |
| `retrieved_kb_evidence` | Top-k KB chunks (evidence_id, score, metadata, text) |

### Assistant content

Strict JSON matching the remediation output schema from `response_schema.py`:
- `executive_summary`, `probable_root_cause`, `scope`, `risk_level`
- `automation_eligibility`, `blast_radius`
- `evidence_ids_used` — must include KB-* and CE-* IDs
- `evidence_from_graph` — KB evidence refs appear before CE-* evidence
- `triage_steps`, `remediation_steps`, `validation_steps`
- `pre_checks`, `post_checks`, `do_not_execute_if`, `rollback_or_safety_notes`
- `escalation_recommendation`, `servicenow_incident_summary`
- `audit_summary`, `confidence_notes`

## Dataset Scale

**Current state:** 4 scenarios (demo-scale, not production-scale).

| Split | Records | Scenarios |
|-------|---------|-----------|
| train | 3 | enterprise_v3_0000, enterprise_v3_0072, enterprise_v3_0073 |
| val   | 1 | enterprise_v3_0074 |

This scale is intentional for the current phase — the dataset validates the pipeline and schema, not model generalisation.

### Scaling guidance

To expand the dataset:
1. Add new enterprise scenarios to `scenario_library/enterprise_gnn_rca/`
2. Add corresponding RCA outputs to `assets/preloaded/enterprise_gnn_rca/`
3. Add domain-specific SOPs to `assets/kb/`
4. Rebuild the KB index
5. Re-run `build_sop_grounded_qwen_training_data.py` with `--scenarios <all_ids>`

For production alignment, aim for:
- 50–200 scenarios per domain
- Multiple SOP variants per domain
- Coverage of edge cases: partial evidence, multi-domain incidents, recovery failures

## Commands

### Generate training data

```bash
# Standard generation (KB index must already be built)
python scripts/build_sop_grounded_qwen_training_data.py --strict-kb --pretty

# Rebuild KB index first, then generate
python scripts/build_kb_index.py --reset
python scripts/build_sop_grounded_qwen_training_data.py --strict-kb --pretty

# Custom KB top-k
python scripts/build_sop_grounded_qwen_training_data.py --kb-top-k 8 --strict-kb

# Specific scenarios only
python scripts/build_sop_grounded_qwen_training_data.py \
    --scenarios enterprise_v3_0072 enterprise_v3_0073 \
    --strict-kb
```

### Validate output

```bash
python - <<'PY'
import json
from pathlib import Path

for p in [
    Path("data/qwen_sop_grounded/train.jsonl"),
    Path("data/qwen_sop_grounded/val.jsonl"),
]:
    print("checking", p)
    lines = p.read_text().strip().splitlines()
    assert lines, f"{p} is empty"
    for line in lines:
        rec = json.loads(line)
        assert len(rec["messages"]) == 3
        json.loads(rec["messages"][1]["content"])
        out = json.loads(rec["messages"][2]["content"])
        assert "evidence_ids_used" in out
        assert any(str(x).startswith("KB-") for x in out["evidence_ids_used"])
    print("PASS")
PY
```

## Integrity Constraints

- `labels.json` is never read. The KB retrieval pipeline reads only RCA JSON and events.json.
- No `expected_root_cause`, `ground_truth_node`, `correct_top1`, `correct_top_k`, `reciprocal_rank`, or `evaluation` fields appear anywhere.
- `target_source: "template_sop_grounded"` — the target is deterministic template output, not model-generated.
- KB-* IDs in `evidence_ids_used` are always traceable to specific KB chunks via `chunk_id = f"{kb_id}::chunk-{i:03d}"`.
- Model retraining is not triggered by this script. This script only generates data.
